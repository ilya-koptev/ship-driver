# Баг homeui: заголовок устройства на странице «Устройства» не обновляется из `/meta` (`title`) при первой загрузке

## Краткое описание

На странице **«Устройства»** (`/#/devices`) заголовок устройства, опубликованный во внешнем
MQTT‑устройстве через `title` в JSON‑объекте `/devices/<id>/meta`, **не отображается при
загрузке страницы по F5** — показывается голый `id` устройства. Имя из `title` появляется
только после **перехода на другую страницу и обратно** 

То есть данные в MQTT корректны и соответствуют конвенции — проблема чисто в реактивности
отрисовки списка устройств.

## Окружение

| | |
|---|---|
| Плата | Wiren Board 8 (`TARGET=wb8/trixie`) |
| Прошивка | `RELEASE_NAME=unstable.latest`, `SUITE=unstable`, сборка `202512290931` |
| `wb-mqtt-homeui` | **2.235.3** |
| `wb-homeui-backend` | **2.235.3** |
| Бандлы фронтенда | `assets/devices-CprAeBwC.js` (модель/стор), `assets/devices-BUEMpUT0.js` (страница) |

На Wiren Board 7 (старый homeui) тот же набор MQTT‑топиков отображается корректно сразу.

## Шаги воспроизведения

1. Внешним публикатором (не wb‑сервис) создать устройство в MQTT, например:

   ```
   # ВАЖНО: исторически контролы этого устройства уже существовали в дереве ретейна
   #        раньше, чем был добавлен JSON-объект /meta с полем title
   mosquitto_pub -r -t /devices/boat4/meta \
     -m '{"driver":"ship-driver","title":{"en":"boat4 (channel 19)","ru":"boat4 (channel 19)"}}'
   mosquitto_pub -r -t /devices/boat4/meta/name -m 'boat4 (channel 19)'
   mosquitto_pub -r -t /devices/boat4/controls/enabled/meta -m '{"type":"switch","readonly":false,"order":1}'
   mosquitto_pub -r -t /devices/boat4/controls/enabled -m '1'
   # ... ещё контролы ...
   ```

2. Открыть `http://<controller>/#/devices`.
3. Нажать **F5** (полная перезагрузка), оставаясь на странице «Устройства».

## Ожидаемое поведение

В заголовке устройства — `boat4 (channel 19)` (из `title`, как у штатных WB‑устройств,
например `wbrules` → «Настройки движка правил»).

## Фактическое поведение

- После **F5**: заголовок = `boat4` (id устройства).
- После **перехода на другую страницу и возврата** :
  заголовок = `boat4 (channel 19)` (корректно).

Поведение **детерминированное** и воспроизводится:
- в обычном и в приватном (incognito) окне — то есть это не кэш браузера;
- после полной перезагрузки контроллера (mosquitto перечитал ретейн) — то есть это не
  «застрявшее» состояние брокера;
- независимо от значения `driver` (`ship-driver`, `wb-rules`) — то есть дело не в распознавании
  драйвера.

Проверено, что MQTT‑данные корректны и структурно идентичны штатным устройствам:

```
/devices/boat4/meta       {"driver":"ship-driver","title":{"en":"boat4 (channel 19)","ru":"boat4 (channel 19)"}}
/devices/wbrules/meta     {"driver":"wb-rules","title":{"en":"Rule engine settings","ru":"Настройки движка правил"}}
```

## Анализ причины (по коду фронтенда)

В `assets/devices-CprAeBwC.js` класс устройства использует **приватные поля** `#t` (переводы
`title`) и `#e` (значение `/meta/name`), а имя вычисляется геттером:

```js
get name(){ return this.#t[n.language] || this.#t.en || this.#e || this.id }
```

`title` сохраняется в `setMeta` (всегда, до проверки драйвера):

```js
setMeta(e){
  try{
    let t = JSON.parse(e);
    if (this.#t = t.title || {}, Object.values(x).includes(this.type)) return;
    t.driver === `wb-rules` ? this.type = x.Virtual
      : (t.driver === `wb-modbus` || t.driver === `wb-mqtt-zigbee`) && (this.type = x.Modbus);
  } catch(e){ console.error(`Invalid meta format:`, e); }
}
```

Обработчики MQTT в сторе:

```js
{ handledTopic: `${r}/meta`,      handler: () => { if(t){ let e=this.#i(n); e.setMeta(t); e.explicit=!0 } ... } }
{ handledTopic: `${r}/meta/name`, handler: () => { if(t){ let e=this.#i(n); e.name=t;    e.explicit=!0 } ... } }
```

**Ключевая проблема** — конструктор делает устройство наблюдаемым с **пустыми аннотациями**:

```js
constructor(t){
  this.id = t;
  this.isVisible = !v().includes(this.id);
  (b(t) || this.isServiceDevice) && (this.type = x.System);
  e(this, {}, { autoBind: true });   // makeObservable(this, {}, …) — аннотации пустые
}
```

mobx **не может** сделать наблюдаемыми приватные поля класса (`#t`, `#e`), и здесь они даже не
указаны в аннотациях. Поэтому геттер `name`, зависящий от `#t`/`#e`, **не реактивен**.

Страница (`assets/devices-BUEMpUT0.js`) рисует заголовок так:

```js
children: [ t.name, t.type === m.Virtual && jsx(S,{className:`devices-icon`}), ... ]
```

### Последовательность, приводящая к багу

1. При загрузке homeui получает ретейн‑пачку. Устройство добавляется в наблюдаемый `Map`
   стора по приходу **первого** своего топика. Если первым приходит `/controls/.../meta`
   (а не `/meta`), мутация `Map` вызывает рендер списка, и в этот момент `#t` ещё пуст →
   заголовок = `id`.
2. Затем приходит `/devices/<id>/meta` → `setMeta` записывает `#t = title`, **но** изменение
   приватного поля не наблюдаемо → перерисовки нет → заголовок остаётся `id`.
3. Имя обновляется только когда компонент списка пересоздаётся (навигация/переключение вкладки),
   так как геттер `name` пересчитывается заново.

Штатные WB‑устройства не страдают, потому что их `/meta` попадает в дерево ретейна раньше их
контролов (исторически создаётся первым) и успевает к первой отрисовке. У внешних устройств,
где контролы появились в дереве раньше, чем `title` был добавлен в `/meta`, гарантировать
порядок доставки ретейна невозможно.

## Предлагаемое исправление

Сделать имя устройства реактивным одним из способов:

1. Перенести `#t`/`#e` в **обычные** (не приватные) наблюдаемые поля и объявить их в аннотациях
   `makeObservable`/`makeAutoObservable` (геттер `name` станет `computed`); **либо**
2. Хранить переводы заголовка в наблюдаемом контейнере (`observable.map`/`observable.ref`) и
   обновлять его в `setMeta`/`set name`.

После этого позднее пришедшая `/devices/<id>/meta` (`title`) будет вызывать перерисовку
заголовка без пересоздания компонента.

## Обходной путь (для пользователей)

Имя из `title` появляется при любой навигации по меню. Голый `id` остаётся только при F5
непосредственно на странице «Устройства». На WB7 (старый homeui) проблемы нет.
