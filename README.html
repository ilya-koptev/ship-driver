Драйвер корабля — ship-driver

Кастомный драйвер корабельной шины для Wiren Board (192.168.100.6). Python-файл + конфиг /etc/ship-driver.conf (редактируется в вебе: Настройки → Конфигурационные файлы → «Ship Driver»).

Что это

Заменяет штатный wb-mqtt-serial + rs485-arbiter на корабельной шине. Владеет всеми четырьмя LoRa-передатчиками /dev/ttyMOD1..4 — это 4 равнозначных канала. На каждом канале независимо крутится один и тот же автомат корабля: опрос по Modbus (pwm8a04 адреса 11/12/13 + ИБП WB-UPS-v3 адрес 10), инжекция кадров плеера DFPlayer прямо в шину, автомат режимов, расписание опроса, ограничение тока заряда по температуре. Всё зеркалится в MQTT (устройства boat1..4 — операционный API + визуализация для стороннего софта).

На одной радио-частоте Modbus-адреса у всех кораблей одинаковые, поэтому одновременно на канале общаемся только с одним кораблём; выбор корабля — по номеру корабля (= LoRa-адрес приёмника). Разводка моторов/света и радио-параметры каждого корабля заданы в конфиге по его номеру: при смене номера на канале драйвер сразу переписывает модем и переключает таблицу регистров на нужный корабль.

Режимы и логика работы

  • SEARCH — корабль не на связи. Канал непрерывно зондирует эфир (ток батареи ~раз в 0.8 с). Поймал ответ → инициализация (частота 400, моторы 40, подсветка 0) → переход в рабочий режим.
  • SAILING — пришла любая команда управления (моторы/подсветка/частота); сбрасывает таймер 30 с. Команды плеера режим НЕ меняют.
  • CHARGING — 30 с без команд и ток батареи > 0 (идёт заряд). Ток заряда 2000 мА; при темп. АКБ > 50 °C снижается до 600 мА, возврат к 2000 при < 48 °C (гистерезис 2 °C: в зоне 48–50 °C уставка не меняется).
  • IDLE — 30 с без команд и ток батареи ≤ 0 (дрейф на батарее).
  • SERVICE — по переключателю service (опрос всего; для отладки/настройки).
  • OFF — канал выключен (enabled=0): порт закрыт, эфир не трогается.
  • Любой режим → SEARCH, если ток батареи перестал читаться (2 промаха × 5 с ≈ 10 с).

По умолчанию все 4 канала включены → пустые каналы сидят в SEARCH, пока на них не появится корабль. Вкл/выкл каналов сохраняется (/etc/ship-driver-state.json) и переживает перезагрузку; настройки модемов при старте читаются из самих модемов.

MQTT — топики

ID устройств: boat1…boat4 (в интерфейсе — «Boat 1…4»). Формат:

  • Чтение значения: /devices/boat<N>/controls/<контрол>
  • Команда (запись): тот же топик + суффикс /on
  • Имя устройства: /devices/boat<N>/meta/name
Контрол (топик внутри …/controls/)ДоступНазначение
enabledЗап.канал вкл/выкл (1/0)
modeЧт.текущий режим (SEARCH/SAILING/CHARGING/IDLE/SERVICE/OFF)
battery_currentЧт.ток батареи, А (>0 = заряд)
battery_temperatureЧт.температура АКБ, °C
charge_levelЧт.заряд, %
back_left / front_left / back_right / front_rightЗап.моторы, 40–80% (40 = холостой)
light1 … light5Зап.подсветка, 0–100% (light4=«Ходовые огни», light5=«Внутренняя подсветка»)
mp3_trackЗап.плеер: 0=стоп, 1–15=трек
mp3_volumeЗап.плеер: громкость 0–30
ship_numberЗап.номер корабля (= LoRa-адрес) — пишется в модем сразу, переключает канал на этот корабль и его разводку

Частота ШИМ, ток заряда и параметры LoRa (канал/air-rate/мощность) на дашборде boatN отсутствуют — они в конфиге. На борту фиксированный набор: 4 мотора + 5 огней.

Примеры (mosquitto):

mosquitto_pub -t /devices/boat1/controls/front_right/on -m 50   # мотор перёд-право = 50%
mosquitto_pub -t /devices/boat1/controls/light1/on      -m 100  # подсветка 1 = 100%
mosquitto_pub -t /devices/boat1/controls/mp3_track/on   -m 3    # играть трек 3
mosquitto_pub -t /devices/boat2/controls/ship_number/on -m 23   # канал 2 -> корабль №23 (пишется сразу)
mosquitto_sub -t /devices/boat1/controls/mode                   # текущий режим
mosquitto_sub -t /devices/boat1/controls/battery_current        # ток батареи

Конфигурация — /etc/ship-driver.conf

Все «крутилки» вынесены в JSON-конфиг, который правится прямо в вебе: Настройки → Конфигурационные файлы → «Ship Driver» (схема wb-mqtt-confed). После сохранения сервис перезапускается автоматически. Две группы:

  • Основное: скорость шины (список), таймаут ответа, заряд (токи и пороги темп.), значения инициализации, пределы интерфейса (моторы/трек), матрица опроса (+ «таймаут без команд» и «промахи до SEARCH»), LoRa-план MOD1–4 (канал/air-rate/адрес/мощность каждого передатчика), «каналы включены при старте».
  • Корабли: «По умолчанию» (общие air_rate+power и резервная разводка) и «Список кораблей» — каждый корабль по своему номеру (=LoRa-адрес): LoRa-канал и разводка моторов/света → ШИМ (адрес+канал).

Канал LoRa выбирается из списка с частотами; разрешённые ГКРЧ помечены ✓ ГКРЧ (CH14/16/17/19), остальные — ⚠ вне ГКРЧ. Несколько параметров захардкожены (база частоты 850.125 МГц, профиль E220, макс. громкость, пределы контролов, дефолты дашборда).

Дашборд «Ship Setup (RS485-1)» — преднастройка корабля по проводу

MQTT-устройство ship_setup для прошивки LoRa-модема корабля через /dev/ttyRS485-1. Редактируется только «Номер корабля»; остальное — кнопки и индикация:

  • service — ON ставит все радио-каналы на паузу (эфир/провод свободны), OFF — возобновляет.
  • ship_number — номер корабля (= LoRa-адрес), единственное редактируемое поле; LoRa_address — его дубль (чтение).
  • «Считать» (LoRa_read) — читает модем подключённого корабля и показывает LoRa_channel / LoRa_air_rate / LoRa_freq и его номер.
  • «Записать» (LoRa_apply) — берёт параметры корабля с этим номером из конфига (раздел «Корабли»: канал + общие air-rate/мощность) и пишет их в модем. Если номера нет в конфиге → ERR корабль №N не настроен.
  • LoRa_status — результат операции.

Порядок: подключить корабль к RS485-1 → его модем в CONFIG (переключателем на плате) → service ON → задать «Номер корабля» → «Записать» (или «Считать» для проверки) → вернуть модем в RUN → service OFF. Канал MODn с тем же каналом/скоростью подхватит корабль по его номеру.

Как переключать корабль на канале

Канал boatN = передатчик MODn на своей частоте (из LoRa-плана). Чтобы переключить канал на другой корабль — задать на boatN контрол «Номер корабля» (ship_number): драйвер сразу пишет этот адрес в модем и переключает разводку моторов/света на конфиг этого корабля. Корабль и «его» канал MOD должны совпадать по каналу и air-rate.

Важно: «онлайн» определяется по току батареи UPS (адрес 10). Если UPS на корабле отключён/обесточен — канал останется в SEARCH, даже если pwm и радио работают.
RU/ГКРЧ-каналы: CH14=864.125, CH16=866.125, CH17=867.125, CH19=869.125 МГц.

Установка / обновление / управление

Файлы на ВБ (ставит пакет): /usr/bin/ship-driver.py, /lib/systemd/system/ship-driver.service, конфиг /etc/ship-driver.conf (conffile — правки переживают обновление), схема веб-редактора /usr/share/wb-mqtt-confed/schemas/ship-driver.schema.json, состояние каналов /etc/ship-driver-state.json (создаётся сам).

Установка (apt-репозиторий на GitHub Pages):

echo "deb [trusted=yes] https://ilya-koptev.github.io/ship-driver ./" | sudo tee /etc/apt/sources.list.d/ship-driver.list
sudo apt update
sudo apt install ship-driver

Обновление:

sudo apt update && sudo apt upgrade

Управление:

systemctl status|restart|stop|start ship-driver
journalctl -u ship-driver -f          # живой лог
systemctl disable|enable ship-driver  # автозапуск
Драйвер при старте сам останавливает wb-mqtt-serial (директивы Conflicts / ExecStartPre), чтобы единолично владеть портом, и сам переводит модем каждого канала в прозрачный (рабочий) режим при открытии порта — для всех MOD1–4. При старте параметры модемов читаются (не пишутся). Зависимости: python3, pyserial, paho-mqtt, mosquitto; для веб-редактора конфига — wb-mqtt-confed.

Временно отдать шину штатному wb-mqtt-serial (напр., для настройки УПС)

Драйвер и wb-mqtt-serial не могут работать одновременно (директива Conflicts) — оба претендуют на один порт. Иногда нужно временно вернуть штатный serial-драйвер (например, чтобы настроить ИБП штатными средствами WB). Делается через start/stop, без enable/disable — автозапуск не трогаем, и после перезагрузки сам вернётся ship-driver.

Отдать шину штатному serial:

systemctl stop  ship-driver
systemctl start wb-mqtt-serial

Вернуть наш драйвер:

systemctl stop  wb-mqtt-serial
systemctl start ship-driver
  • Ручной systemctl stop ship-driver честно гасит драйвер — Restart=always здесь не мешает (он про падения, а не про ручной стоп).
  • Из-за Conflicts переключение фактически делает и одна команда: systemctl start wb-mqtt-serial сам погасит драйвер, а systemctl start ship-driver — погасит serial. Пары stop+start выше делают это явно.
  • Автозапуск не меняем → после ребута снова поднимется ship-driver. Если настройка затянется через перезагрузку — тогда уже systemctl disable --now ship-driver / enable --now wb-mqtt-serial, и обратно симметрично.

Проверить, что активно: systemctl is-active ship-driver wb-mqtt-serial.
