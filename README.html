Драйвер корабля — ship-driver

Кастомный драйвер корабельной шины для Wiren Board (192.168.100.6). Один Python-файл.

Что это

Заменяет штатный wb-mqtt-serial + rs485-arbiter на корабельной шине. Владеет всеми четырьмя LoRa-передатчиками /dev/ttyMOD1..4 — это 4 равнозначных канала. На каждом канале независимо крутится один и тот же автомат корабля: опрос по Modbus (pwm8a04 адреса 11/12/13 + ИБП WB-UPS-v3 адрес 10), инжекция кадров плеера DFPlayer прямо в шину, автомат режимов, расписание опроса, ограничение тока заряда по температуре, а также настройка собственного LoRa-модема канала. Всё зеркалится в MQTT (устройства Ship 1..4).

На одной радио-частоте Modbus-адреса у всех кораблей одинаковые, поэтому одновременно на канале общаемся только с одним кораблём; выбор корабля — по LoRa-адресу (перенастройкой передатчика канала).

Режимы и логика работы

  • SEARCH — корабль не на связи. Канал непрерывно зондирует эфир (ток батареи ~раз в 0.8 с). Поймал ответ → инициализация (частота 400, моторы 40, подсветка 0) → переход в рабочий режим.
  • SAILING — пришла любая команда управления (моторы/подсветка/частота); сбрасывает таймер 30 с. Команды плеера режим НЕ меняют.
  • CHARGING — 30 с без команд и ток батареи > 0 (идёт заряд). Ток заряда 2000 мА; при темп. АКБ > 50 °C снижается до 300 мА, возврат к 2000 при < 48 °C.
  • IDLE — 30 с без команд и ток батареи ≤ 0 (дрейф на батарее).
  • SERVICE — по переключателю service (опрос всего; для отладки/настройки).
  • OFF — канал выключен (enabled=0): порт закрыт, эфир не трогается.
  • Любой режим → SEARCH, если ток батареи перестал читаться (2 промаха × 5 с ≈ 10 с).

По умолчанию все 4 канала включены → пустые каналы сидят в SEARCH, пока на них не появится корабль. Состояние каналов (вкл/выкл + конфиг модема) сохраняется и переживает перезагрузку.

MQTT — топики

ID устройств: boat1…boat4 (в интерфейсе — «Boat 1…4»). Формат:

  • Чтение значения: /devices/boat<N>/controls/<контрол>
  • Команда (запись): тот же топик + суффикс /on
  • Имя устройства: /devices/boat<N>/meta/name

Примеры (mosquitto):

mosquitto_pub -t /devices/boat1/controls/front_right/on -m 50   # мотор перёд-право = 50%
mosquitto_pub -t /devices/boat1/controls/light1/on      -m 100  # подсветка 1 = 100%
mosquitto_pub -t /devices/boat1/controls/mp3_track/on   -m 3    # играть трек 3
mosquitto_pub -t /devices/boat2/controls/LoRa_address/on -m 23   # LoRa-адрес канала 2 -> 23 (применяется сразу)
mosquitto_sub -t /devices/boat1/controls/mode                   # текущий режим
mosquitto_sub -t /devices/boat1/controls/battery_current        # ток батареи

Дашборд настройки — «Ship Setup (RS485-1)»

Отдельное MQTT-устройство ship_setup для проводной преднастройки корабля по /dev/ttyRS485-1. Корабль подключается к RS485-1, его LoRa-модем переводится в config-режим (переключателем на плате).

  • service — переключатель: ON ставит все радио-каналы на паузу (mode SERVICE, эфир/провод свободны), OFF — возобновляет.
  • LoRa_channel / LoRa_air_rate / LoRa_address / LoRa_power + кнопки LoRa_read / LoRa_apply + LoRa_status — чтение/запись LoRa-модема корабля по проводу (как у каналов, но через RS485-1; модем корабля должен быть в config).

Порядок: подключить корабль к RS485-1 → модем в config → service ON → задать LoRa_address (уникальный) и канал/скорость/мощность под нужный MOD → LoRa_apply → вернуть модем в RUN → service OFF. Нужный MOD-канал поймает корабль по LoRa-адресу.

Важно: «онлайн» определяется по току батареи UPS (адрес 10). Если UPS на корабле отключён/обесточен — канал останется в SEARCH, даже если pwm и радио работают.

Как настраивать LoRa

Принцип связи: канал boat1..4 = один передатчик MODn на своей частоте. Корабль ловит тот канал, у которого совпадают частота (channel) и скорость (air_rate); адрес различает корабли на одной частоте (на одной частоте активен один корабль на канал). Под выбором канала серым показана итоговая частота LoRa_freq (МГц).

1) Передатчик канала (MOD1–4) — на устройстве boatN:

  1. Задать LoRa_channel (под ним обновится LoRa_freq, МГц), LoRa_air_rate (kbps), LoRa_power (dBm).
  2. Нажать LoRa_apply — параметры записываются в модем; LoRa_status покажет результат (OK ch=.. freq=.. air=.. addr=.. power=..).
  3. LoRa_address применяется сразу при изменении — так быстро переключают канал на другой корабль (по его LoRa-адресу).
  4. LoRa_read — прочитать текущие настройки модема MODn.

2) Преднастройка корабля по проводу — дашборд «Ship Setup (RS485-1)»:

  1. Подключить корабль к разъёму RS485-1; перевести его LoRa-модем в режим CONFIG (переключателем на плате корабля).
  2. Включить service на дашборде — все радио-каналы встанут на паузу (эфир и провод свободны).
  3. Задать LoRa_channel (видно частоту), LoRa_air_rate, уникальный LoRa_address, LoRa_power → LoRa_apply (LoRa_read — проверить). Запись идёт по проводу, поэтому модем должен быть в CONFIG.
  4. Вернуть LoRa-модем корабля в RUN (transparent), выключить service.
  5. Канал MODn с такой же частотой/скоростью подхватит корабль. Нужен включённый UPS на корабле — «онлайн» определяется по пульсу тока батареи; без UPS канал останется в SEARCH.
Правило согласования: у корабля и «его» канала MOD должны совпадать channel и air_rate (а в transparent-режиме адрес для связи не важен — он нужен, чтобы различать корабли при работе в fixed-режиме). RU-каналы: CH14=864.125, CH16=866.125, CH17=867.125, CH19=869.125 МГц.

Установка / обновление / управление

Файлы на ВБ: /usr/local/bin/ship-driver.py, /etc/systemd/system/ship-driver.service, состояние /etc/ship-driver-state.json (создаётся сам).

Установка с нуля:

scp ship-driver.py       root@192.168.100.6:/usr/local/bin/
scp ship-driver.service  root@192.168.100.6:/etc/systemd/system/
ssh root@192.168.100.6 'chmod +x /usr/local/bin/ship-driver.py; systemctl daemon-reload; systemctl enable --now ship-driver'

Обновление:

scp ship-driver.py root@192.168.100.6:/usr/local/bin/
ssh root@192.168.100.6 systemctl restart ship-driver

Управление:

systemctl status|restart|stop|start ship-driver
journalctl -u ship-driver -f          # живой лог
systemctl disable|enable ship-driver  # автозапуск
Драйвер при старте сам останавливает wb-mqtt-serial (директивы Conflicts / ExecStartPre), чтобы единолично владеть портом, и сам переводит модем каждого канала в прозрачный (рабочий) режим при открытии порта — для всех MOD1–4. Зависимости (уже стоят): python3, pyserial, paho-mqtt, mosquitto.

Временно отдать шину штатному wb-mqtt-serial (напр., для настройки УПС)

Драйвер и wb-mqtt-serial не могут работать одновременно (директива Conflicts) — оба претендуют на один порт. Иногда нужно временно вернуть штатный serial-драйвер (например, чтобы настроить ИБП штатными средствами WB). Делается через start/stop, без enable/disable — автозапуск не трогаем, и после перезагрузки сам вернётся ship-driver.

Отдать шину штатному serial:

systemctl stop  ship-driver
systemctl start wb-mqtt-serial

Вернуть наш драйвер:

systemctl stop  wb-mqtt-serial
systemctl start ship-driver
  • Ручной systemctl stop ship-driver честно гасит драйвер — Restart=always здесь не мешает (он про падения, а не про ручной стоп).
  • Из-за Conflicts переключение фактически делает и одна команда: systemctl start wb-mqtt-serial сам погасит драйвер, а systemctl start ship-driver — погасит serial. Пары stop+start выше делают это явно.
  • Автозапуск не меняем → после ребута снова поднимется ship-driver. Если настройка затянется через перезагрузку — тогда уже systemctl disable --now ship-driver / enable --now wb-mqtt-serial, и обратно симметрично.

Проверить, что активно: systemctl is-active ship-driver wb-mqtt-serial.
