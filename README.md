# ship-driver

Кастомный драйвер корабельной шины для **Wiren Board**. Владеет всеми четырьмя
LoRa‑передатчиками `/dev/ttyMOD1..4` (4 равнозначных канала); на каждом независимо крутится один
и тот же автомат корабля: опрос по Modbus (`pwm8a04` адреса 11/12/13 + ИБП `WB‑UPS‑v3` адрес 10),
инжекция кадров плеера DFPlayer прямо в шину, автомат режимов, расписание опроса, ограничение тока
заряда по температуре. Всё зеркалится в MQTT — устройства `boat1..4`.

Кроме того драйвер обслуживает **беспроводные зарядные станции** `charger1..N`: коммутирует реле
передатчика и магнитов фиксации и меряет ток передатчика — по Modbus RTU через TCP‑шлюз (EBYTE
serial server), независимо от корабельной LoRa‑шины.

Заменяет на корабельной шине штатный `wb-mqtt-serial` + `rs485-arbiter`.

**Документ разбит на два блока:**

- **[🛰 Удалённое управление по MQTT](#-удалённое-управление-по-mqtt)** — для тех, кто управляет
  кораблём и зарядками со стороннего софта или дашборда: топики, контролы, команды. Железо знать
  не обязательно.
- **[🔧 Устройство, монтаж и настройка](#-устройство-монтаж-и-настройка)** — для того, кто
  собирает, ставит и настраивает драйвер на контроллере: принцип работы, подключение, конфиг,
  установка, релиз.

Полная MQTT‑спецификация с примерами — **[`MQTT-API.md`](MQTT-API.md)**. Спецификация компонентов
корабля (BOM) — **[`BUILD.md`](BUILD.md)**.

# 🛰 Удалённое управление по MQTT

Всё, что нужно для удалённого управления; вникать в железо и установку не обязательно.

**Соглашение по топикам:** чтение — `/devices/<устройство>/controls/<контрол>`, команда — тот же
топик с суффиксом `/on`. Устройства: `boat1..4` (корабли), `charger1..N` (зарядки) и `ship_setup`
(проводная преднастройка — относится к блоку настройки).

## Корабли `boat1..4`

Драйвер создаёт устройства `boatN` **только для подключённых каналов** (порт `/dev/ttyMODn` есть и
LoRa‑модем отвечает). Набор на борту фиксированный: 4 мотора + 5 огней. Пока канал не на связи
(режим SEARCH) или выключен (OFF), на дашборде остаются только `enabled`, `mode` и `ship_number`;
остальные контролы убираются и возвращаются при выходе в онлайн.

| Контрол | Доступ | Назначение |
|---|---|---|
| `enabled` | зап. | канал вкл/выкл (1/0) |
| `mode` | чт. | текущий режим (см. ниже) |
| `battery_current` / `battery_temperature` / `charge_level` | чт. | телеметрия ИБП |
| `input_voltage` | чт. | входное напряжение ИБП, В (опрос 5 с, вместе с током) |
| `back_left` / `front_left` / `back_right` / `front_right` | зап. | моторы (Back/Front × Left/Right), 40–80 % (40 = холостой) |
| `nav_lights` / `morse_lamp` / `deck_lights` / `cabin_light1` / `cabin_light2` | зап. | свет, 0–100 % |
| `mp3_track` | зап. | плеер: 0 = стоп, 1–15 = трек |
| `mp3_volume` | зап. | плеер: громкость 0–30 |
| `ship_number` | зап. | номер корабля (= LoRa‑адрес) — переключает канал на этот корабль и переинициализирует его |

Частота ШИМ, ток заряда и параметры LoRa (канал/air‑rate/мощность) на дашборде `boatN` отсутствуют
— они в конфиге контроллера.

```sh
mosquitto_pub -t /devices/boat1/controls/front_right/on -m 50   # мотор перёд-право = 50%
mosquitto_pub -t /devices/boat1/controls/nav_lights/on  -m 100  # ходовые огни = 100%
mosquitto_pub -t /devices/boat1/controls/mp3_track/on   -m 3    # играть трек 3
mosquitto_pub -t /devices/boat2/controls/ship_number/on -m 23   # канал 2 -> корабль №23 (сразу)
mosquitto_sub -t /devices/boat1/controls/mode                   # текущий режим
```

### Режимы (`mode`)

| Режим | Когда | Что делает |
|---|---|---|
| **SEARCH** | корабль не на связи | непрерывно зондирует эфир; поймал ответ → инициализация → рабочий режим |
| **SAILING** | пришла любая команда управления | сбрасывает таймер 30 с (команды плеера режим не меняют) |
| **CHARGING** | 30 с без команд и ток батареи > 0 | ток заряда 2000 мА; при t АКБ > 50 °C → 600 мА, возврат к 2000 при < 48 °C (гистерезис 2 °C) |
| **IDLE** | 30 с без команд и ток ≤ 0 | дрейф на батарее |
| **OFF** | `enabled = 0` | порт закрыт, эфир не трогается |

Любой режим → **SEARCH**, если корабль перестал отвечать (2 промаха). По умолчанию все 4 канала
включены; вкл/выкл каналов **и заданный `ship_number`** переживают перезагрузку контроллера.

### Номер корабля (`ship_number`)

На одной радио‑частоте Modbus‑адреса у всех кораблей одинаковые, поэтому на канале одновременно
общаемся только с одним кораблём — его различает **номер корабля** (= LoRa‑адрес приёмника). Запись
в `ship_number` мгновенно переключает канал на нужный корабль: драйвер пишет адрес в модем,
подхватывает разводку моторов/света этого корабля из конфига и **переинициализирует его** (частота
ШИМ + холостой газ) через короткий SEARCH. Без этой переинициализации при двух кораблях на одном
канале второй корабль мог не получить `FREQ`/холостой газ, и его моторы не запускались (свет при
этом работал).

### Матрица опроса (период чтения телеметрии)

| Параметр | SEARCH | SAILING | IDLE | CHARGING |
|---|---|---|---|---|
| Ток батареи + вх. напряжение ИБП | ~0.8 с | 2 с | 5 с | 5 с |
| Температура АКБ | — | 60 с | 60 с | 10 с |
| Процент заряда | — | 60 с | 60 с | 20 с |
| Моторы + свет (обратная сверка, блочное чтение) | — | 30 с | 60 с | 30 с |
| Проверка частоты ШИМ (авто-восстановление 400 Гц) | — | 10 с | 10 с | 10 с |

Команды управления (запись моторов/света/плеера) идут немедленно, вне расписания.

## Зарядки `charger1..N`

Береговые беспроводные зарядные станции. Каждая появляется в MQTT как устройство `charger<N>`
(заголовки `Charger 1…N`). Управление — теми же `/on`‑командами.

| Контрол | Доступ | Назначение |
|---|---|---|
| `transmitter` | зап. | передатчик зарядки вкл/выкл (1/0) |
| `magnets` | зап. | магниты фиксации судна вкл/выкл (1/0) |
| `transmitter_current` | чт. | ток передатчика, А |

```sh
mosquitto_pub -t /devices/charger1/controls/transmitter/on -m 1   # передатчик вкл
mosquitto_pub -t /devices/charger1/controls/magnets/on     -m 1   # магниты вкл
mosquitto_sub -t /devices/charger1/controls/transmitter_current   # ток передатчика
```

> При старте драйвер только **отражает** фактическое состояние реле зарядки, ничего не переключая.

# 🔧 Устройство, монтаж и настройка

Настройки вынесены в **`/etc/ship-driver.conf`** (JSON) и правятся прямо в веб‑интерфейсе WB:
**Настройки → Конфигурационные файлы → «Ship Driver»** (схема `wb-mqtt-confed`). После сохранения
сервис перезапускается автоматически.

## Принцип работы

Корабли на канале различаются **номером корабля** (= LoRa‑адрес приёмника, см.
[«Номер корабля»](#номер-корабля-ship_number)). При старте драйвер приводит модем к конфигу
(**config‑authoritative**): читает регистры и, если отличаются, пишет полный дамп (адрес/канал/
air‑rate/мощность + reg5). Адрес берётся из сохранённого `ship_number` (иначе из конфига),
канал/air‑rate/мощность — из конфига. Адрес можно менять на лету контролом `ship_number`.

«Онлайн» = ответил **хотя бы один** из 4 Modbus‑устройств: UPS по току (адрес 10) **или** любой
pwm8a04 по регистру частоты (11/12/13). Поэтому корабль находится **даже при выключенном UPS**
(по pwm) — только телеметрия батареи будет пустой.

Инициализация при выходе в онлайн: мощность 0 → частота ШИМ 400 → моторы/свет из конфига. Вкл/выкл
каналов и `ship_number` сохраняются в `/etc/ship-driver-state.json`.

**Самопроверка частоты ШИМ** (группа `freq_check`, каждые 10 с): читается частота каналов pwm8a04; если
она сбросилась в дефолт (не 400 Гц) — драйвер переписывает 400 и пишет в журнал (`freq drift … re-set`).
Часто (10 с) — чтобы после **перезагрузки pwm‑модуля по броску тока** (моторы дают просадку) частота
восстанавливалась за секунды. Это ловит случай, когда pwm‑модуль моргнул питанием и потерял частоту,
оставшись «онлайн» — тогда моторы не запускаются, а свет работает; раньше это лечилось только уходом в
SEARCH или сменой `ship_number`.
Обратное чтение моторов и света (сверка с записанным) идёт **блочно** — один Modbus‑обмен на модуль
(регистры 112–114), а не поканально.

## Подключение света (выходы платы OUT1–OUT5)

На плате пять световых **выходов** `OUT1…OUT5`, которые коммутирует сборка Дарлингтона
**ULN2003A** (U6, ключи с открытым коллектором — «втекающий» ток, замыкают выход на землю).
ШИМ 5 В с каналов `pwm8a04` идёт на входы ULN. Нагрузку (лента/лампа) подключают между
**общим +12 В платы** и нужным выходом `OUTx`. «+12 В» выведено на тех же выходных разъёмах
рядом с выходами; общий вывод ULN (COM) заведён на +12 В (для защитных диодов).

Свет разведён через **три модуля `pwm8a04`** (Modbus‑адреса `a11 / a12 / a13`) и **два выходных
разъёма** платы:

> ⚡ **Выходы 4 и 5 — сдвоенные:** на каждый параллельно работают **два** ключа ULN, поэтому они
> держат примерно **вдвое больший ток** — ставь на них более мощные нагрузки.

**Токи (ULN2003A):**

- Выходы **1–3** (одиночный ключ) — до **~500 мА** длительно (пик 600 мА).
- Выходы **4 и 5** (два ключа параллельно) — до **~1 А**.
- Напряжение нагрузки до 50 В (на плате +12 В); падение на ключе `VCE(sat)` ≈ 1.1–1.6 В (греется).
- **Суммарно по корпусу** ULN мощность ограничена (≈ 1 Вт при 25 °C): все выходы на максимуме
  одновременно держать нельзя — действует деретинг по числу одновременно открытых ключей и
  температуре. ШИМ со скважностью < 100 % снижает средний нагрев.

**Как пользоваться:** «+» нагрузки — на **+12 В** платы, «−» — на нужный выход. Яркость задаётся
ШИМ 0–100 % через контрол света (`nav_lights` … `cabin_light2`). Мощную нагрузку бери на
сдвоенные выходы **4/5**; если нужно ещё больше — внешний MOSFET.

## Зарядные станции — железо

Береговая станция стоит на **Modbus‑RTU‑over‑TCP шлюзе** (прозрачный EBYTE serial server,
`host:port`), независимо от корабельной LoRa‑шины. Железо: передатчик беспроводной зарядки
**XKT‑801** и **магниты фиксации** судна — на **разных каналах** реле **WB‑MRM2‑mini** (реле NC,
поэтому в конфиге стоит «инверсия»); ток передатчика снимается как падение на шунте и меряется
модулем **WB‑MAI6** (ток = напряжение / сопротивление шунта).

Несколько станций могут висеть на одном шлюзе (делят TCP‑соединение) либо каждая на своём; один
WB‑MAI6 может обслуживать несколько станций (разные входы `IN`). Управление по MQTT — см. блок
[«Зарядки `charger1..N`»](#зарядки-charger1n).

## Конфигурация — `/etc/ship-driver.conf`

Правится в вебе (**Настройки → Конфигурационные файлы → «Ship Driver»**) либо файлом. Разделы:

- **Основное:** скорость шины (список), таймаут ответа, заряд (токи и пороги темп.),
  значения инициализации, пределы интерфейса (моторы/трек), матрица опроса (+ «таймаут без
  команд» и «промахи до SEARCH»), «каналы включены при старте».
- **Радио:** план MOD1–4 — канал / air‑rate / адрес / мощность каждого передатчика.
- **Корабли:** «По умолчанию» (общие `air_rate` + `power` и резервная разводка) и «Список
  кораблей» — каждый по своему **номеру (= LoRa‑адрес)**: LoRa‑канал и разводка моторов/света
  → ШИМ (адрес + канал).
- **Зарядки:** список зарядных станций. У каждой — свой TCP‑шлюз (`host:port`), выходы реле
  **«Передатчик»** и **«Магниты фиксации»** (адрес реле + канал K + инверсия для NC) и датчик
  тока **WB‑MAI6** (адрес + вход `IN` + сопротивление шунта). Имена устройств — `Charger N`.

Канал LoRa выбирается из списка с частотами; разрешённые ГКРЧ помечены `✓ ГКРЧ`
(CH14/16/17/19), остальные — `⚠ вне ГКРЧ`. Захардкожены: база частоты 850.125 МГц, профиль
E220 (UART 9600, subpkt128, RSSI), макс. громкость, пределы контролов, дефолты дашборда.
RU/ГКРЧ‑каналы: CH14 = 864.125, CH16 = 866.125, CH17 = 867.125, CH19 = 869.125 МГц.

## Дашборд «Ship Setup (RS485‑1)» — преднастройка корабля по проводу

MQTT‑устройство `ship_setup` для прошивки LoRa‑модема корабля через `/dev/ttyRS485-1`.
Редактируются **«Номер корабля» (`ship_number`)** и **«LoRa канал» (`LoRa_channel`)**; остальное —
индикация и кнопки:

- `ship_number` / `LoRa_channel` — задаются вручную; `LoRa_address` — дубль номера (чтение);
  `LoRa_freq` и `Band (GKRCh)` — частота и диапазон по выбранному каналу (пересчитываются сразу);
- **«Read»** (`LoRa_read`) — читает 9 байт регистров подключённого модема и показывает все
  декодированные параметры: адрес, канал, air‑rate, мощность, LBT, UART, субпакет, RSSI эфира/в
  пакете, режим TX, WOR, версия, `raw (9 bytes)` + строка `LoRa default` для сравнения;
- **«Write»** (`LoRa_apply`) — пишет в модем заданные номер и канал (полный дамп регистров);
- результат операции (OK/ошибка) — в журнале: `journalctl -u ship-driver`.

Порядок: подключить корабль к RS485‑1 → его модем в **CONFIG** (переключателем на плате) →
задать «Номер корабля» и «LoRa канал» → **«Write»** (или «Read» для проверки) → вернуть модем
в **RUN**. Канал MODn с тем же каналом/скоростью подхватит корабль.

## Временно отдать шину штатному `wb-mqtt-serial` (напр., для настройки УПС)

**Проще всего — переключателем на дашборде.** Пакет ставит правило wb-rules
(`/etc/wb-rules/ship-bus-switch.js`) с виртуальным устройством **«Драйвер шины корабля»**
(`ship_bus`): переключатель `ship_driver` (вкл = `ship-driver`, выкл = `wb-mqtt-serial`) и
индикатор активного сервиса. Логика идемпотентна — перезагрузка движка wb-rules сервисы не дёргает.
При **остановке** драйвера (переключении на serial) он сам убирает свои дашборды `boatN` и
`Ship Setup` (чистит retained‑топики); при старте объявляет их заново.

Либо вручную через `systemctl`. Драйвер и `wb-mqtt-serial` не работают одновременно
(`Conflicts`). Временно — через `start`/`stop`, **без** `enable/disable` (после ребута сам
вернётся `ship-driver`):

```sh
systemctl stop  ship-driver  && systemctl start wb-mqtt-serial   # отдать serial
systemctl stop  wb-mqtt-serial && systemctl start ship-driver    # вернуть драйвер
```

Из‑за `Conflicts` достаточно и одной команды старта нужного сервиса. Если настройка затянется
через перезагрузку — тогда `systemctl disable --now ship-driver` / `enable --now wb-mqtt-serial`.

## Установка на контроллер (один раз)

> **Подготовка железа.** Каждый MOD‑слот с LoRa‑модулем нужно настроить в конфигурации
> Wiren Board как тип **«LoRa»** (веб: Настройки → Конфигурация Wiren Board → MODx → LoRa).
> Без этого UART слота не поднимается — `/dev/ttyMODn` не открывается (ошибка
> *Could not configure port: Input/output error*), и канал считается отсутствующим
> (дашборд `boatN` не появляется).

Перед установкой не помешает обновиться.

```sh
echo "deb [trusted=yes] https://ilya-koptev.github.io/ship-driver ./" | sudo tee /etc/apt/sources.list.d/ship-driver.list
sudo apt update
sudo apt install ship-driver
```

Ставит: драйвер `/usr/bin/ship-driver.py`, юнит `/lib/systemd/system/ship-driver.service`,
конфиг `/etc/ship-driver.conf` (dpkg **conffile** — правки переживают обновление), схему
веб‑редактора `/usr/share/wb-mqtt-confed/schemas/ship-driver.schema.json`; тянет зависимости
(`python3-serial`, `python3-paho-mqtt`, `mosquitto`; рекомендует `wb-mqtt-confed`); включает и
запускает сервис. Ручную установку (`/usr/local/bin/ship-driver.py` + юнит в
`/etc/systemd/system/`) пакет при установке удаляет.

## Сбор логов (дашборд «Сбор логов корабля»)

Если с кораблём случилась непонятная ситуация — в homeui есть устройство **«Сбор логов корабля»**
(`ship_diag`, ставится пакетом, wb-rules): выбираешь **корабль** (Все / номер из настроенных) и
**период в часах**, жмёшь **«Собрать логи»**. Драйвер собирает архив и в поле **«Результат»** даёт
ссылку на скачивание из браузера.

Что попадает в `/var/www/ship-logs-<корабль>-<время>.tar.gz`:
- журнал `ship-driver` за период (события, ошибки, `freq drift`, смены `ship_number`, charger);
- **телеметрия из wb-mqtt-db** по бортам (CSV: режим/ток/напряжение/заряд/углы по времени);
- `ship-driver.conf` + `ship-driver-state.json` + версия пакета + снимок текущих значений из MQTT.

Скачать: открыть страницу `http://<контроллер>/ship-logs.html` — список архивов с кликабельными
ссылками (обновляется после каждого сбора; та же авторизация, что и homeui). Хранятся
последние 10 архивов. Можно и вручную: `ship-collect-logs <номер|all> <часов>`.

## Доступ и роли

Веб-панель работает под логином. Роли:
- **operator** — дашборд «Ship Driver» и устройства: управление бортами, зарядкой, сбор логов; разделы настроек скрыты.
- **admin** — то же плюс настройки/конфиги (правка `/etc/ship-driver.conf`: модули LoRa, зарядки, матрица опроса).

Пользователи и роли заводятся в веб-панели (раздел настроек). Для оператора управление и телеметрия
не меняются — отличается только доступ к настройкам (они под `admin`).

Внешнее управление по MQTT — порт `1883`, ограничен топиками корабля (`boat*` / `charger*` / `ship_*`);
прочие топики контроллера на этом порту закрыты. Подробнее — в [MQTT-API.md](MQTT-API.md).

## Релиз новой версии (мейнтейнеру)

1. Правишь драйвер (`ship-driver.py`) / схему (`ship-driver.schema.json`) / конфиг / доки.
2. **Поднимаешь `VERSION`** (apt обновляет только на бóльшую версию).
3. `git commit` + `git push` в `main`.
4. GitHub Action **build-and-publish-apt** собирает `ship-driver_<VERSION>_all.deb`,
   пересобирает apt‑репозиторий и публикует на GitHub Pages.
5. На контроллере: `sudo apt update && sudo apt upgrade`.

> ⚠️ Для перепубликации делай **новый коммит** (или запусти workflow через *workflow_dispatch*).
> **Не** жми «Re-run jobs» на старом прогоне — появятся дубли артефактов `github-pages`, и
> `deploy-pages` упадёт с *«Multiple artifacts named github-pages»*.

## Настройка репозитория (один раз, на GitHub)

- Репозиторий **public** (для GitHub Pages + apt по https без авторизации).
- Settings → **Pages** → Source = **GitHub Actions**.

apt‑репозиторий сейчас **без подписи** (`[trusted=yes]`). Для GPG‑подписи — подписывать
`Release` в workflow и поставлять публичный ключ.
