# База знаний: замеры, диагностика, гочи

Живые наблюдения и разборы инцидентов по ship-driver. Отдельно от README (там — как пользоваться);
здесь — **что измерено на борту и почему что-то ломалось**, чтобы не переоткрывать заново.

Все электрические замеры сделаны на живом борте (корабль №10, контроллер .105, канал MOD4),
ток снимался с WB-UPS (регистр Ibat, разряд — отрицательный). **Питание:** батарея ~8 В (Ibat — её ток), УПС бустит в 12-В шину, DC-DC делает 5 В — на них сидят И моторы (BLDC-драйверы), И логика (pwm8a04/LoRa/DFPlayer). Мощность = **Ibat × Vbat (~8 В)**, НЕ ×12.

## 1. Потребление и связь (замерено)

### Холостой ход и связь по LoRa
| Состояние | Ток батареи | Примечание |
|---|---|---|
| Дрейф, без опроса | **−114…−116 мА** (~1.4 Вт) | «чистый» холостой — только электроника |
| Под непрерывным опросом | −133 мА | +19 мА даёт сам обмен по шине |

- **Транзакция по LoRa T:** одиночная запись ≈ **75 мс**, блочное чтение 3 регистров ≈ **80 мс**.
- **Ёмкость шины:** ≈ **13 транзакций/с**.
- **Soak (60 с и 210 с непрерывного опроса):** 0 ошибок, дрейфа задержки нет (p95 ≈ 81 мс, max ≈ 81 мс).
- **Цена непрерывного опроса на пределе:** +18…19 мА к разряду — пренебрежимо.
- **Вывод:** частоту опроса выбираем по полезности (курс, безопасность), а не ради экономии.

### Моторы (главный потребитель)
Все 4 мотора вместе, Δ — прибавка к холостому 40:

| Скважность | Ток батареи | Δ (моторы) | Кратно связи (~19 мА) |
|---|---|---|---|
| 43 % | −331 мА | ~200 мА | ×10 |
| 45 % | −544 мА | ~410 мА | ×22 |
| 47 % | −795 мА | ~660 мА | ×35 |
| 50 % | −1.25 А | ~1.12 А | ×59 |

Два мотора (back_left + back_right), шаг 2 %:

| Скв. | 42 | 44 | 46 | 48 | 50 | 52 | 54 |
|---|---|---|---|---|---|---|---|
| Δ, мА | 97 | 152 | 280 | 400 | 550 | 716 | **916** |

- **1 А в батарее** на двух моторах достигается при **54 %** скважности.
- ~**275 мА на мотор** при 50 %; масштабируется **линейно по числу моторов**.
- Рост **резко нелинейный по газу**: 42→54 % (всего +12 п.п.) → ток моторов ×9.4.
- **Вывод:** тяга — единственный значимый потребитель (на 1–2 порядка выше связи); дальность = дисциплина газа, а не экономия на телеметрии.

### Мощность моторов (Ibat×Vbat) и порог отруба
Борт на магнитах, заряд off, 40→60 % шаг 2 % (1 мотор = back_right, 2 = BL+BR, 4 = все).
**Холостой (40 %) вычтен** — в ячейке только **приращение от моторов** в виде **напряжение · ток = мощность** (Vbat под нагрузкой · ΔIbat = ΔP; напряжение замеренное, не вычитается).
Базовый холостой (взведённый, 40 %): ≈ **0.10 А / 0.8 Вт** при Vbat ≈ 7.8 В — **подтверждён историей `wb-mqtt-db`** (см. ниже), duty 0 и 40 не отличаются.

|  %  | 1 мотор | 2 мотора | 4 мотора |
|---|---|---|---|
|     | В · А = Вт | В · А = Вт | В · А = Вт |
| 40 | 7.82 · 0 = 0 | 7.79 · 0 = 0 | 7.76 · 0 = 0 |
| 42 | 7.81 · 0.03 = 0.2 | 7.78 · 0.08 = 0.6 | 7.75 · 0.12 = 0.9 |
| 44 | 7.80 · 0.06 = 0.5 | 7.78 · 0.14 = 1.1 | 7.74 · 0.20 = 1.5 |
| 46 | 7.79 · 0.11 = 0.9 | 7.75 · 0.25 = 1.9 | 7.71 · 0.38 = 2.9 |
| 48 | 7.79 · 0.16 = 1.2 | 7.74 · 0.36 = 2.8 | 7.69 · 0.54 = 4.2 |
| 50 | 7.78 · 0.21 = 1.6 | 7.71 · 0.49 = 3.8 | 7.66 · 0.75 = 5.7 |
| 52 | 7.76 · 0.28 = 2.2 | 7.69 · 0.64 = 4.9 | 7.62 · 0.97 = 7.4 |
| 54 | 7.74 · 0.35 = 2.7 | 7.66 · 0.83 = 6.4 | **браунаут** (7.50 В, ток скакнул к ~1.7 А) |
| 56 | 7.74 · 0.41 = 3.2 | 7.62 · 0.98 = 7.5 | **браунаут** (7.43 В, ~2.1 А) |
| 58 | 7.72 · 0.49 = 3.8 | 7.60 · 1.14 = 8.7 | **overcurrent** |
| 60 | 7.71 · 0.58 = 4.5 | 7.56 · 1.38 = 10.4 | **overcurrent** |

**Аддитивность (в линейной зоне, ~50–52 %):** пара ≈ **2×** одиночного (напр. 52 %: 2.2 → 4.9 Вт), а квадро — заметно **меньше 4×** (52 %: 7.4 Вт ≈ 3.4× одиночного, ~1.5× пары), т.е. по числу моторов приращение **сублинейно**. По газу — наоборот, резко **нелинейно** (у порога ток взлетает).

**Порог отруба (все 4) плавает с зарядом.** В этом прогоне (батарея уже подсажена тестом) 4 мотора устойчивы до **52 % (≈7.4 Вт, ток АКБ ~1.07 А, Vbat 7.62 В)**; на **54 %** Vbat просела к **7.50 В** и ток скакнул к ~1.7 А — начался **браунаут**. В более раннем прогоне на свежей батарее держалось до 56 % (~11 Вт). На 57 %+ модули pwm8a04 **ребутятся** (журнал: `freq drift 20000/1000`); с **1.9.6** freq_check при таком дрейфе форсирует реинициализацию → борт уходит в холостой (40) и переармится, **разрывая петлю браунаута** (до 1.9.6 переписывалась только частота, борт стартовал и снова браунаутил).

**Механизм — по напряжению, не по току.** Отруб = когда **Vbat под нагрузкой садится к ~7.4–7.5 В**: 12-В шина/DC-DC не держат 5 В → сброс pwm (моторы и логика на одной 5-В шине). Внутреннее сопротивление АКБ+провода ≈ **0.15–0.17 Ом** (Vbat 7.76→7.43 В при росте тока на ≈2 А). ⚠ Порог по ТОКУ зависит от заряда: на **разряженной** батарее Vbat стартует ниже → отрубит при **меньшем** токе/газе (этот прогон — прямая иллюстрация: то же железо отрубилось на 54 %, а не 56 %).

**Выводы:** железо — узкое место 5-В DC-DC (моторы+логика вместе), усилить/развести; алгоритм — не гнать все 4 разом в потолок (и помнить, что потолок падает с разрядом). `freq_check` (раз в 10 с) ловит сброс модуля и с 1.9.6 форсирует полную реинициализацию (переарм ESC + холостой), а не только частоту.

### duty 0 не экономит
Скважность 0 против «взведённого» холостого 40: разница ~**5 мА** (в пределах шума).
ESC остаются под питанием — обнулением газа холостой ток **не срезать**.
Чтобы экономить в дрейфе — снимать питание **целых узлов** (силовая часть ESC, DFPlayer, подсветка), а не крутить duty.

### Из чего складываются ~115 мА холостого (ОЦЕНКА, не поканальный замер)
| Узел | @12 В | |
|---|---|---|
| 4× BLDC-драйверы (standby) | ~40–60 мА | обычно самый жирный кусок |
| 3× pwm8a04 | ~20–30 мА | |
| Модуль ИБП (Modbus addr 10) | ~15–25 мА | |
| LoRa E220 (RX) | ~8–10 мА | слушает эфир постоянно |
| DFPlayer mini | ~8–10 мА | |
| RS485 + потери DC-DC | ~5–10 мА | |

## 2. Разборы инцидентов

### 16.07 — «свет работает, моторы нет»
**Причина:** `init_ship` (ставит freq=400 и взводит ESC) выполнялся только на переходе SEARCH→online,
но **не при смене `ship_number`**. Второй корабль на общем канале не получал частоту/холостой → моторы мертвы, свет жив.
**Фикс:** 1.7.1 — реинициализация при смене `ship_number`; плюс группа **`freq_check`** (раз в 10 с) —
самолечение случая «PWM ребутнулся по току на ходу». В 1.7.1 она чинила только частоту; с **1.9.6**
форсирует полную реинициализацию (переарм ESC), т.к. одной частоты для оживления моторов мало (см. 24.07).

### 22.07 13:48 — «не реагирует, перезагрузил через панель»
**Причина:** борт был **оффлайн по радио** (сошёл с зарядки + ребут). Пока `self.online == False`,
**все команды моторов/света/mp3 молча отбрасываются** (`ship-driver.py`, требуется `self.online`) — без лога и без ошибки в топике.
В журнале ровно в 13:44 местного (`10:44 UTC`) — `init_ship ship=10`: это момент возврата на связь.
SEARCH очень частый (0.05 с), поэтому восстановление почти мгновенное, как только вернулась связь.
**Слабое место:** нет обратной связи — оператор не отличает «борт вне связи» от «драйвер завис».

### 22.07 14:02 — «из ПО не проходит, из панели оживает»
**Причина (по коду):** драйвер подписан **только** на командный топик `…/controls/+/on`.
Если внешнее ПО пишет в топик значения `…/controls/<имя>` (без `/on`) — драйвер это **не слышит**; панель WB всегда пишет в `/on` → работает.
**Проверить на стороне ПО:** публикация должна идти в `/devices/boat4/controls/<имя>/on`.
Побочно: значение мотора клампится в 40–80.

### 24.07 — «в сбросе и не очухивается» после теста порога 57 %+
**Симптом:** после прогона порога отруба (моторы браунаутили на 57–60 %) борт остался «живой»
(online, freq=400, скважность 40), но на команды моторов **не реагировал** — тот же класс, что 16.07.
**Причина:** браунаут ребутит модули pwm8a04, при этом **ESC теряют арминг**. Борт при этом не уходил
оффлайн (UPS/другой pwm отвечали), поэтому `init_ship` (единственное место, где ESC вооружаются)
**не переотрабатывал**. А `freq_check` восстанавливал **только частоту**, не арминг → холостой/частота
в норме, но регуляторы не вооружены → моторы молчат. Ложный сигнал: BR=55 давал Ibat ≈ 0 —
это была **маскировка входом** (была включена зарядка), а не «мотор мёртв».
**Диагностика/лечение вручную:** `enabled` 0→1 (форс оффлайн→online→`init_ship`) переармил ESC.
Чистая проверка (зарядка OFF): BR=55 → −0.47 А, FL=55 → −0.49 А — оба мотора ожили.
**Фикс (1.9.6):** `freq_check`, обнаружив дрейф частоты (= модуль ребутнулся), теперь не переписывает
freq, а **форсирует полную реинициализацию** (`online=False` → SEARCH → `init_ship`): freq=400 **+ переарм
ESC + холостой 40**. Борт **сам** восстанавливается после браунаута и «мягко» сбрасывается в холостой
(разрывая петлю браунаута), без ручного `enabled` off/on. ⚠ Оператор увидит кратковременный SEARCH и
уход моторов в холостой — это ожидаемо: после просадки борт возвращается в безопасное состояние, газ надо задать заново.

### Баг конфигуратора — «Сохранить» не работает в списке кораблей
**Причина:** при добавлении корабля JSON-Editor подставляет в поля без `default` нули, а у `pin.slave` стоит `minimum: 1`.
Новый корабль рождается с 9 каналами (4 мотора + 5 огней), у всех `slave = 0` → **9 ошибок валидации** →
confed глушит **глобальную** кнопку «Сохранить». Ошибки не видны — блоки «Моторы»/«Подсветка» свёрнуты.
Существующий конфиг валиден, поэтому ломается только при добавлении/правке.
**Фикс (рекомендуемый):** задать элементу `ships.list.items` **`default` с типовой разводкой** — новый корабль появляется валидным и предзаполненным.
**Обход:** разводка у всех одинакова → неизвестный адрес обслуживается по `ships.default` (`wiring_for()` → fallback), записи в списке сейчас не обязательны; либо править `/etc/ship-driver.conf` напрямую.

## 3. Справочник (как устроено, гочи)

- **Гейтинг команд:** моторы/свет/mp3 применяются **только при `self.online == True`**; оффлайн → тихо игнор. `enabled=False` → принудительный оффлайн (режим OFF).
- **online/offline:** ставится True, когда ответит UPS **или** любой pwm8a04; уходит в оффлайн после `OFFLINE_FAILS` неответов подряд — **без логов**. `search_period = 0.05 с`.
- **`init_ship`** (freq=400 + моторы в холостой + свет 0 + **арминг ESC**) — только на переходе оффлайн→online. С 1.9.6 `freq_check` при сбросе модуля **сам** гонит борт в оффлайн, чтобы этот переход (и переарм) повторился.
- **Режимы:** SEARCH / SAILING / CHARGING / IDLE / SERVICE / OFF. Онлайн: `current > 0 → CHARGING`, иначе IDLE.
- **`ship_number` == LoRa-адрес.** `wiring_for(address)` для неизвестного адреса берёт `ships.default` (сейчас у всех бортов разводка одинаковая).
- **Диапазон газа:** клампится в **40–80**; 40 = взведённый холостой.
- **Часы контроллера — UTC.** Жалобы обычно в местном (UTC+3): вычитай 3 часа при сверке с журналом.
- **Развёрнуто на .105 — 1.7.0** (без 1.7.1-фикса реинициализации и без dev-надстроек: freq_check, поджатая матрица, дашборд логов).

### Доступ и методика замеров
- Путь: `plink` → бридж (`100.98.33.26`, Tailscale) → `sshpass` → `.105`.
- Тестовые скрипты писать в файл и гонять через `plink -m` (инлайн-питон ломается квотингом PowerShell).
- Любой тест с моторами/остановкой драйвера — под `trap "systemctl start ship-driver" EXIT` и с гарантированным возвратом моторов в 40.
- `python3 -u` — иначе буферизация теряет вывод при обрыве по таймауту.

### История значений (`wb-mqtt-db`) — источник истины, а не догадки
- WB пишет телеметрию в **`/var/lib/wirenboard/db/data.db`** (сервис `wb-mqtt-db`, конфиг `/etc/wb-mqtt-db.conf`: интервал 120 с, при неизменном значении — 1200 с).
- Читать через **`wb-mqtt-db-cli --from "YYYY-MM-DD HH:MM:SS" --to ... --time-format "%H:%M:%S" boat4/battery_current`** (время **UTC**). Отдаёт бакеты `avg/min/max`.
- Гоча: `--decimal-places` **роняет** CLI (`TypeError`); `sqlite3` на контроллере **нет** — только CLI.
- **Зачем:** ловит артефакты замеров. Пример: `measure3` показал холостой L=40 = **222 мА**, а история за те же сутки — стабильно **~110 мА** (и в паузах старого прогона `max` = −0.08…−0.14 А). Т.е. 222 мА — артефакт разового замера; правильный холостой ≈ **0.10 А** (сходится с разделом 1). Прежде чем строить выводы на разовом замере — сверься с историей.
